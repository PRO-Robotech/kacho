# ServiceAccount

**ServiceAccount** — машинный (non-human) субъект платформы: идентичность для рабочих нагрузок,
CI-пайплайнов, интеграций и автоматизации. Там, где User представляет человека, ServiceAccount
представляет программу. Ему выдают роли ровно так же, как пользователю (через AccessBinding), но
аутентифицируется он не логином, а **OAuth 2.0-ключом** (см. [Токены доступа](/api/tokens)).

Сервис-аккаунт принадлежит аккаунту; опционально может быть привязан к проекту (`projectId`).
Флаг `enabled` позволяет временно отключить возможность аутентификации без удаления.

## Методы

`ServiceAccountService` — стандартный CRUD (`Create`/`Update`/`Delete` async; `Get`/`List` sync)
+ журнал операций. Выпуск ключей — отдельный сервис `SAKeyService` (см.
[Токены доступа](/api/tokens)).

### Get — получить сервис-аккаунт

<ApiOperation method="GET" endpoint="/iam/v1/serviceAccounts/{service_account_id}">

Возвращает сервис-аккаунт по идентификатору. Требует `v_get`.

<CodeBlock language="json">
  {dedent`
    {
      "id": "sva_9q2r5t8k1m4p7",
      "accountId": "acc_2k9f3qw8x7m2n",
      "name": "ci-builder",
      "enabled": true,
      "createdAt": "2026-06-24T10:10:00Z"
    }
  `}
</CodeBlock>

</ApiOperation>

### List — список сервис-аккаунтов

<ApiOperation method="GET" endpoint="/iam/v1/serviceAccounts">

Возвращает сервис-аккаунты аккаунта, **видимые вызывающему** (default-deny через scope-filter,
не 403). Обязателен `accountId`; поддерживает `filter` по `name`.

<CodeBlock language="bash">
  {dedent`
    curl 'http://localhost:18080/iam/v1/serviceAccounts?accountId=acc_2k9f3qw8x7m2n&pageSize=50' \\
      -H 'Authorization: Bearer <JWT>'
  `}
</CodeBlock>

</ApiOperation>

### Create — создать сервис-аккаунт

<ApiOperation method="POST" endpoint="/iam/v1/serviceAccounts" async>

Создаёт сервис-аккаунт в аккаунте. Требует `editor` на аккаунте. Асинхронно
(`response: ServiceAccount`).

<CodeBlock language="bash">
  {dedent`
    curl -X POST 'http://localhost:18080/iam/v1/serviceAccounts' \\
      -H 'Authorization: Bearer <JWT>' \\
      -H 'Content-Type: application/json' \\
      -d '{ "accountId": "acc_2k9f3qw8x7m2n", "name": "ci-builder" }'
  `}
</CodeBlock>

</ApiOperation>

### Update — изменить сервис-аккаунт

<ApiOperation method="PATCH" endpoint="/iam/v1/serviceAccounts/{service_account_id}" async>

Меняет `name` / `description` / `labels` по `updateMask`. `accountId` immutable. Требует
`v_update`. Асинхронно.

<CodeBlock language="bash">
  {dedent`
    curl -X PATCH 'http://localhost:18080/iam/v1/serviceAccounts/sva_9q2r5t8k1m4p7' \\
      -H 'Authorization: Bearer <JWT>' \\
      -H 'Content-Type: application/json' \\
      -d '{ "updateMask": "description", "description": "CI builder (main pipeline)" }'
  `}
</CodeBlock>

</ApiOperation>

### Delete — удалить сервис-аккаунт

<ApiOperation method="DELETE" endpoint="/iam/v1/serviceAccounts/{service_account_id}" async>

Удаляет сервис-аккаунт. Если на него ссылается активный AccessBinding — удаление блокируется
`FAILED_PRECONDITION` (сначала снять привязки). Требует `v_delete`. Асинхронно.

</ApiOperation>

### ListOperations — операции над сервис-аккаунтом

<ApiOperation method="GET" endpoint="/iam/v1/serviceAccounts/{service_account_id}/operations">

Возвращает операции над сервис-аккаунтом. Sync, курсорная пагинация. Требует `v_list`.

</ApiOperation>

:::tip Как SA получает токен
Сервис-аккаунт аутентифицируется через OAuth 2.0-ключ (`SAKeyService.Issue`) и Ory Hydra:
рабочая нагрузка подписывает `client_assertion` приватным ключом и обменивает его в
`/oauth2/token` на kacho-JWT. Детали и федерация — [Токены доступа](/api/tokens).
:::
