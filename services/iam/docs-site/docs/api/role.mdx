# Role

**Role** — именованный набор прав: *что можно делать*. Роль описывает разрешённые действия
(«глаголы») над типами ресурсов («ресурсы»), опционально суженные до конкретных экземпляров.
Роль сама по себе никому доступ не даёт — она становится действующей, только когда её привязывают
к субъекту через [AccessBinding](/api/access-binding). Разделение «роль (что) ↔ привязка (кому и
где)» — основа модели: одну роль переиспользуют для многих субъектов и на разных уровнях иерархии.

Роли двух видов:

- **system-роли** (`isSystem=true`) — базовый каталог (`viewer` / `editor` / `admin` и т.п.),
  seed'ятся миграциями, привязаны к кластеру (`cluster_kacho_root`), **read-only** через
  публичный API (Update/Delete → `FAILED_PRECONDITION`);
- **custom-роли** (`isSystem=false`) — создаются тенантом; ровно один из `accountId` /
  `projectId` задан (взаимоисключающе, XOR): роль либо account-scoped, либо project-scoped.

## Модель правил

Политика роли задаётся полем **`rules[]`** (1–64 правила). Каждое `Rule` — однородный грант ровно
**одного модуля**:

<table>
  <thead><tr><th>Поле Rule</th><th>Тип</th><th>Описание</th></tr></thead>
  <tbody>
    <tr><td><code>module</code></td><td>string</td><td>Ровно один domain-модуль: <code>iam</code> / <code>vpc</code> / <code>compute</code> / <code>loadbalancer</code></td></tr>
    <tr><td><code>resources</code></td><td>string[]</td><td>Типы ресурсов модуля (1–16), lowercase (<code>networks</code>, <code>subnets</code>, ...)</td></tr>
    <tr><td><code>verbs</code></td><td>string[]</td><td>Глаголы (1–16): <code>get</code> / <code>list</code> / <code>create</code> / <code>update</code> / <code>delete</code></td></tr>
    <tr><td><code>resourceNames</code></td><td>string[]</td><td>Опц. пин по id (≤256). Взаимоисключающе с <code>matchLabels</code></td></tr>
    <tr><td><code>matchLabels</code></td><td>map</td><td>Опц. отбор по меткам (AND-equality, ≤16). Взаимоисключающе с <code>resourceNames</code></td></tr>
  </tbody>
</table>

Форма правила задаёт «прицел» (anchor):

- **без селекторов** — грант на *все* экземпляры `{module} × resources` в пределах scope привязки;
- **`resourceNames`** — пин на конкретные id (любой тип можно пиновать по id);
- **`matchLabels`** — отбор по меткам (только для label-selectable типов; иначе
  `INVALID_ARGUMENT`).

Роль, покрывающая несколько модулей, выражается несколькими правилами (по одному на модуль).
`rules[]` — источник истины и публичная поверхность роли; клиенты рендерят роль именно из него.
Грантабельную таксономию (какие модули/ресурсы/глаголы допустимы) отдаёт
[PermissionCatalog](/api/authorize#permissioncatalog).

:::note Wildcard `"*"` — только для system-ролей
Literal `"*"` в `resources` / `verbs` / `module` допустим лишь в system-ролях. В custom-роли
он вернёт `INVALID_ARGUMENT` (глагол-`*` — «все глаголы типа» — грантабелен, но модуль/ресурс-`*`
system-only).
:::

## Методы

`RoleService` — CRUD (`Create`/`Update`/`Delete` async; `Get`/`List` sync) + журнал операций.
Мутации применимы только к custom-ролям.

### Get — получить роль

<ApiOperation method="GET" endpoint="/iam/v1/roles/{role_id}">

Возвращает роль по идентификатору. Каталог ролей — tenant-wide reference: чтение доступно любому
аутентифицированному вызывающему (без per-role scope).

<CodeBlock language="json">
  {dedent`
    {
      "id": "rol_6t4v9x2k8m1p5",
      "accountId": "acc_2k9f3qw8x7m2n",
      "name": "vpc-viewer",
      "description": "Read-only VPC networks",
      "isSystem": false,
      "rules": [
        { "module": "vpc", "resources": ["networks"], "verbs": ["get", "list"] }
      ],
      "resourceVersion": "8821",
      "createdAt": "2026-06-24T10:15:00Z"
    }
  `}
</CodeBlock>

</ApiOperation>

### List — каталог ролей

<ApiOperation method="GET" endpoint="/iam/v1/roles">

Возвращает каталог ролей. Поддерживает `filter` (`is_system` / `name`) и `accountId`: при
заданном `accountId` ответ содержит system-роли (видны всегда) плюс custom-роли этого аккаунта.

<CodeBlock language="bash">
  {dedent`
    # только system-роли
    curl 'http://localhost:18080/iam/v1/roles?filter=is_system%3Dtrue' \\
      -H 'Authorization: Bearer <JWT>'
  `}
</CodeBlock>

</ApiOperation>

### Create — создать custom-роль

<ApiOperation method="POST" endpoint="/iam/v1/roles" async>

Создаёт custom-роль. Ровно один из `accountId` / `projectId` (XOR) — иначе `INVALID_ARGUMENT`.
Политика задаётся `rules[]`. Имя, совпадающее с system-ролью, → `ALREADY_EXISTS`. Требует
`editor` на аккаунте роли. Асинхронно (`response: Role`).

<CodeBlock language="bash">
  {dedent`
    curl -X POST 'http://localhost:18080/iam/v1/roles' \\
      -H 'Authorization: Bearer <JWT>' \\
      -H 'Content-Type: application/json' \\
      -d '{
            "accountId": "acc_2k9f3qw8x7m2n",
            "name": "compute-operator",
            "description": "Manage compute instances in prod project",
            "rules": [
              { "module": "compute", "resources": ["instances"], "verbs": ["get","list","update"] }
            ]
          }'
  `}
</CodeBlock>

</ApiOperation>

### Update — изменить custom-роль

<ApiOperation method="PATCH" endpoint="/iam/v1/roles/{role_id}" async>

Меняет `name` / `description` / `rules` / `labels` по `updateMask`. При изменении `rules`
обязателен `resourceVersion` (OCC под конкуренцией; mismatch → `FAILED_PRECONDITION`). Поля
`isSystem` / `accountId` / `projectId` / `clusterId` immutable. **System-роль → `FAILED_PRECONDITION`**.
Требует `v_update`. Асинхронно.

<CodeBlock language="bash">
  {dedent`
    curl -X PATCH 'http://localhost:18080/iam/v1/roles/rol_6t4v9x2k8m1p5' \\
      -H 'Authorization: Bearer <JWT>' \\
      -H 'Content-Type: application/json' \\
      -d '{
            "updateMask": "rules",
            "resourceVersion": "8821",
            "rules": [
              { "module": "vpc", "resources": ["networks","subnets"], "verbs": ["get","list"] }
            ]
          }'
  `}
</CodeBlock>

</ApiOperation>

### Delete — удалить custom-роль

<ApiOperation method="DELETE" endpoint="/iam/v1/roles/{role_id}" async>

Удаляет custom-роль. **System-роль → `FAILED_PRECONDITION`**. Custom-роль с активными
привязками → `FAILED_PRECONDITION` (сначала снять привязки). Требует `v_delete`. Асинхронно.

</ApiOperation>

### ListOperations — операции над ролью

<ApiOperation method="GET" endpoint="/iam/v1/roles/{role_id}/operations">

Возвращает операции над ролью. Sync. Требует `v_list`.

</ApiOperation>

## Ошибки и граничные случаи

<table>
  <thead><tr><th>Ситуация</th><th>Код</th><th>Поведение</th></tr></thead>
  <tbody>
    <tr><td>Get/Update/Delete несуществующей роли</td><td><code>NOT_FOUND</code></td><td><code>"Role &lt;id&gt; not found"</code></td></tr>
    <tr><td>Update / Delete system-роли</td><td><code>FAILED_PRECONDITION</code></td><td>System-роли read-only публично</td></tr>
    <tr><td>Delete custom-роли с активными привязками</td><td><code>FAILED_PRECONDITION</code></td><td>Сначала снять все AccessBinding</td></tr>
    <tr><td>Create: оба / ни одного из <code>accountId</code>/<code>projectId</code></td><td><code>INVALID_ARGUMENT</code></td><td>Scope XOR</td></tr>
    <tr><td>Update <code>rules</code> без / с устаревшим <code>resourceVersion</code></td><td><code>FAILED_PRECONDITION</code></td><td>OCC-конфликт</td></tr>
    <tr><td><code>matchLabels</code> на не-label-selectable типе</td><td><code>INVALID_ARGUMENT</code></td><td><code>"type %s is not selectable (no resource feed)"</code></td></tr>
    <tr><td>Wildcard <code>"&#42;"</code> в custom-роли</td><td><code>INVALID_ARGUMENT</code></td><td>Module/resource-<code>&#42;</code> — system-only</td></tr>
  </tbody>
</table>

:::tip Что можно грантить — спрашивайте у каталога
Список грантабельных модулей/ресурсов, закрытый набор глаголов и wildcard-политику отдаёт
`PermissionCatalogService.ListPermissionCatalog` (`GET /iam/v1/permissionCatalog`). Редактор
ролей должен строиться на нём, а не на захардкоженной копии таксономии. См.
[Авторизация и каталог](/api/authorize#permissioncatalog).
:::
