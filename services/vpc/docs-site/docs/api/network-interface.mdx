# NetworkInterface

**NetworkInterface** (NIC) — это самостоятельный сетевой интерфейс с собственным
жизненным циклом, который Kachō вводит как полноценный ресурс, **отвязанный от какой-либо
рабочей нагрузки**. Вы создаете NIC, конфигурируете его адресацию и набор
security-групп один раз — и дальше управляете «сетевой личностью» (стабильный MAC,
закрепленные IP, политики доступа) независимо от того, что эту личность потребляет. Это
закрывает типичную боль: сетевая идентичность не должна пересоздаваться каждый раз, когда
меняется потребитель.

Поскольку NIC живет отдельно, его адреса и MAC **стабильны**: интерфейс можно создать
заранее, сконфигурировать предсказуемые IP и security-группы, а его MAC аллоцируется
control-plane при создании и не меняется на протяжении всей жизни ресурса. Так вы
получаете воспроизводимую, заранее описанную сетевую конфигурацию вместо набора
эфемерных, привязанных к нагрузке настроек.

NIC принадлежит подсети (`subnetId`; FK `ON DELETE RESTRICT` — пока существует NIC, его
подсеть и транзитивно сеть удалить нельзя). Он несет ссылки на `Address`-ресурсы
(`v4AddressIds` / `v6AddressIds`) и набор security-групп (`securityGroupIds`).

Единая ресурсная модель адресации: **каждый IP — это `Address`-ресурс** (как `Network ←
Subnet ← Address`); NIC лишь «потребляет» их по id. Чтобы выделить новый IP — сначала
создайте internal `Address` (`AddressService.Create` с `internalIpv4AddressSpec.subnetId`)
и передайте его id в `v4AddressIds` / `v6AddressIds`.

:::info Кардинальность адресов
На одной NIC — **максимум один IPv4 и максимум один IPv6** (всего ≤ 2 адреса). Несколько
IP на нагрузку реализуется через **несколько NIC**, а не secondary-адреса в одном
интерфейсе. Ограничение enforced и sync-валидацией, и DB-level
`CHECK (jsonb_array_length(...) <= 1)`.
:::

раскрываются (см. [Авторизация и приватность](/architecture/authz)).

## Create

## Типичные сценарии

- **Предсказуемая сетевая идентичность.** Заранее создаете NIC с закрепленным `Address` и
  набором security-групп — стабильный MAC и IP позволяют завязать на них правила доступа
  и мониторинг, не пересоздавая интерфейс при смене потребителя.
- **Несколько IP — несколько NIC.** Нужна нагрузка с несколькими адресами? Создаете по
  интерфейсу на адрес (на одном NIC — ≤1 v4 и ≤1 v6) и инвентаризуете их через `List`
  с фильтром `subnetId` / `networkId`.
- **Централизованная политика доступа.** Меняете `securityGroupIds` интерфейса через
  `Update` — политика применяется к NIC, его адресам и MAC без переадресации IP.
- **Аудит.** `ListOperations` дает хронологию изменений конкретного интерфейса.

## Подводные камни и рекомендации

:::caution Что важно знать
- **Адреса — отдельные ресурсы.** NIC хранит **ссылки** (`v4AddressIds` /
  `v6AddressIds`), а не сами IP. Перед созданием/обновлением убедитесь, что `Address`
  существует, свободен и принадлежит той же подсети (`subnetId`), что и NIC.
- **Кардинальность жесткая.** ≤1 IPv4 и ≤1 IPv6 на интерфейс — enforced и
  sync-валидацией, и DB-level `CHECK`. Попытка добавить второй адрес того же семейства
  будет отклонена.
- **MAC неизменен.** `macAddress` аллоцируется при `Create`, уникален в пределах облака и
  стабилен всю жизнь NIC — клиент задать или сменить его не может.
- **`subnetId` immutable.** Перенести интерфейс в другую подсеть нельзя — создайте новый
  NIC в нужной подсети.
- **NIC держит подсеть.** FK `ON DELETE RESTRICT`: пока интерфейс существует, его подсеть
  (и сеть) удалить не получится. Порядок удаления снизу вверх: NIC → Address → Subnet →
  Network.
- **Удаление освобождает адреса.** При `Delete` адреса возвращаются в IPAM. Если адрес
  должен пережить интерфейс — управляйте им как самостоятельным `Address`.
:::

:::tip Best practices
- Описывайте NIC, его `Address` и security-группы как единый, заранее подготовленный блок
  инфраструктуры — это дает воспроизводимую и неэфемерную сетевую конфигурацию.
- Все мутации асинхронные: дожидайтесь `done: true` в `Operation`, прежде чем считать
  интерфейс готовым.
- Для инвентаризации используйте `List` с `subnetId` / `networkId` и осмысленные
  `labels` (`role`, `team`) — это основной способ навигации по интерфейсам проекта.
:::
