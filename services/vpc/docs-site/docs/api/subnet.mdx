# Subnet

**Subnet** — это диапазон адресов внутри сети, из которого выделяются внутренние IP.
Подсеть отвечает на практический вопрос «откуда брать адреса для ресурсов»: вы задаете
один или несколько CIDR-блоков (IPv4 и/или IPv6), и из них автоматически выделяются
внутренние адреса (`Address`) и сетевые интерфейсы (`NetworkInterface`). Так вы сами
планируете адресацию приложения — разбиваете сеть на сегменты под разные зоны, ярусы
или команды.

Способ размещения подсети задается дискриминатором `placementType` (обязателен и
immutable):

- **ZONAL** — подсеть живет в одной зоне доступности (`zoneId`); адреса unicast,
  привязаны к зоне. Классический сегмент под ресурсы конкретной зоны.
- **REGIONAL** — подсеть region-scoped (`regionId`); ее префикс анонсируется
  active-active из здоровых зон региона (anycast). Используется, когда адрес должен быть
  доступен из всего региона без привязки к одной зоне (например, региональный VIP
  network load balancer).

Ровно одно из полей `zoneId` / `regionId` заполнено — в зависимости от `placementType`.
Подсеть принадлежит одной виртуальной сети (`networkId`). Платформа гарантирует, что
**CIDR-блоки подсетей одной сети не пересекаются** (контроль на уровне БД,
EXCLUDE-constraint) — это исключает «двойную адресацию» и связанные с ней инциденты еще до
того, как адрес будет кому-то выдан. Подсеть может быть привязана к таблице маршрутизации
(`routeTableId`); если ее не указать — выполняется авто-ассоциация с самой ранней
`RouteTable` сети.

`networkId`, `placementType`, `zoneId` и `regionId` — **immutable** после создания
(сегмент нельзя «переехать» в другую сеть, зону, регион или сменить тип размещения). Набор
CIDR-блоков меняется **не** через `Update` (там эти поля soft-immutable, no-op), а через
специальные действия `:add-cidr-blocks` / `:remove-cidr-blocks` — отдельные операции, чтобы
расширение/сжатие адресного пространства было явным и проходило проверку на пересечения.

:::info Идентификатор и размещение
ID подсети — префикс `sub` + 17 символов crockford-base32 (например,
`sub-7k2m9p4q1r8s3t6vw`). Зона (`zoneId`) и регион (`regionId`) — ресурсы отдельного домена
`kacho-geo`; их существование проверяется при `Create` синхронным вызовом `kacho-geo`
(несуществующая зона/регион → `InvalidArgument "unknown zone id '<id>'"` /
`"unknown region id '<id>'"`, недоступность домена → `Unavailable`).
:::

В теле `Update` поля `v4_cidr_blocks` / `v6_cidr_blocks` принимаются в `update_mask`, но
их применение — **no-op** (запрос завершается успешно, реального изменения набора CIDR

`update_mask` → `InvalidArgument`.
:::

## Create

<ApiOperation method="POST" endpoint="/vpc/v1/subnets" async>

Создает подсеть в указанной сети. Возвращает `Operation` (async). `placementType`
обязателен и определяет пару полей размещения: **ZONAL** → задается `zoneId` (а `regionId`
должен быть пуст); **REGIONAL** → задается `regionId` (а `zoneId` должен быть пуст).
Пропущенный/`UNSPECIFIED` `placementType`, лишнее поле размещения и несуществующая
зона/регион отклоняются **синхронно** (`InvalidArgument`); несуществующая сеть, дубль имени
и пересечение CIDR — тоже синхронно (`NOT_FOUND "Network %s not found"` / `ALREADY_EXISTS` /
`FAILED_PRECONDITION "Subnet CIDRs can not overlap"`). В worker'е: проверка существования
проекта → вставка Subnet (FK + EXCLUDE + CHECK-размещения — атомарный backstop) → запись
событий в outbox.

## Сценарии использования

- **Планирование адресации по зонам (ZONAL).** Разбейте сеть на подсети — по одной на
  зону — с непересекающимися диапазонами (`10.0.0.0/24`, `10.0.1.0/24`, …). Из них
  платформа сама раздает внутренние адреса интерфейсам ресурсов.
- **Региональный сегмент (REGIONAL).** Создайте подсеть с `placementType=REGIONAL` и
  `regionId`, когда адрес должен быть доступен из всего региона без привязки к одной зоне —
  например, под региональный VIP network load balancer (префикс анонсируется active-active
  из здоровых зон региона).
- **Расширение без простоя.** Когда диапазона перестает хватать, добавьте новый CIDR-блок
  через `:add-cidr-blocks` — это штатный путь увеличить адресное пространство подсети, не
  пересоздавая ее и не трогая уже выданные адреса.
- **Dual-stack (IPv4 + IPv6).** Задайте и `v4CidrBlocks`, и `v6CidrBlocks` — интерфейсы
  смогут получать адреса обоих семейств (≤ 1 v4 и ≤ 1 v6 на интерфейс).
- **Аудит занятости.** `ListUsedAddresses` показывает, какие адреса заняты и какие ресурсы
  на них ссылаются (`references`) — удобно перед сжатием диапазона или удалением подсети.
- **DHCP-параметры.** Через `dhcpOptions` (`domainName`, `domainNameServers[]`,
  `ntpServers[]`) задаются сетевые настройки, раздаваемые ресурсам подсети.

## Подводные камни

:::caution Пересечение CIDR
CIDR-блоки подсетей одной сети не должны пересекаться. Конфликт ловится на уровне БД
(EXCLUDE-constraint) и при `Create` / `:add-cidr-blocks` возвращается синхронно как
`FAILED_PRECONDITION "Subnet CIDRs can not overlap"`. Блоки внутри одного запроса тоже не
должны перекрывать друг друга.
:::

:::caution Удаление непустой подсети
Подсеть нельзя удалить, пока в ней есть внутренние `Address` или `NetworkInterface` — FK
`ON DELETE RESTRICT` отдает `FAILED_PRECONDITION`. Сначала освободите ресурсы в порядке
`NetworkInterface → Address`, затем удаляйте подсеть.
:::

:::caution placement_type задается один раз
`placementType` обязателен при `Create` (`UNSPECIFIED` → `InvalidArgument`, в `ZONAL` не
дефолтит) и immutable. Для `ZONAL` заполняется `zoneId`, а `regionId` должен быть пуст; для
`REGIONAL` — наоборот. Лишнее поле размещения → `InvalidArgument`
(`"region_id must be empty for ZONAL placement"` / `"zone_id must be empty for REGIONAL
placement"`). Сменить тип размещения нельзя — создайте новую подсеть.
:::

:::caution host-биты CIDR обязаны быть нулевыми
Адрес сети должен быть выровнен по маске: `10.0.0.0/24` — корректно, `10.0.0.5/24` →
`InvalidArgument`. Это относится и к `Create`, и к `:add-cidr-blocks`.
:::

:::note CIDR не меняются через Update
В `Update` поля `v4CidrBlocks` / `v6CidrBlocks` принимаются в `updateMask`, но их
применение — **no-op** (запрос успешен, набор CIDR не меняется). Единственный способ
изменить диапазоны — `:add-cidr-blocks` / `:remove-cidr-blocks`. Поля `networkId`,
`placementType`, `zoneId`, `regionId` — hard-immutable: явное указание в `updateMask` →
`InvalidArgument`.
:::

## Рекомендации

- **Закладывайте запас, но не дробите чрезмерно.** Выбирайте размер блока с учетом роста
  числа интерфейсов в зоне; расширяться позже можно через `:add-cidr-blocks`.
- **Непересекающаяся раскладка заранее.** Спланируйте диапазоны подсетей одной сети так,
  чтобы они гарантированно не накладывались — это избавит от `FAILED_PRECONDITION` при
  массовом создании.
- **Освобождайте перед сжатием.** Перед `:remove-cidr-blocks` убедитесь через
  `ListUsedAddresses`, что в удаляемом диапазоне нет занятых адресов.
