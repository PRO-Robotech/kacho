# Как участвовать в разработке kacho-vpc

Спасибо за интерес к проекту. Этот документ описывает, как сообщить о проблеме,
подготовить изменение и провести его через ревью.

## Сообщить об ошибке или предложить улучшение

Все задачи ведутся в **GitHub Issues**:
`https://github.com/PRO-Robotech/kacho-vpc/issues`.

- **Баг** — приложите версию, шаги воспроизведения, ожидаемое и фактическое
  поведение, релевантные логи. Метка `bug`.
- **Улучшение / фича** — опишите задачу и зачем она нужна (какую проблему решает),
  до обсуждения дизайна. Метка `enhancement`.
- Перед заведением проверьте, нет ли уже похожего issue.

Не используйте Issues для вопросов по безопасности — см. раздел ниже.

## Окружение

- Go 1.25+, Postgres 16+, Docker (для integration-тестов на testcontainers).
- Сборка и проверки:

```bash
make build          # bin/kacho-vpc
make build-migrator # bin/kacho-migrator
make test-short     # быстрые тесты (моки)
make test           # полный прогон с integration (нужен Docker)
make vet
make lint
```

## Поток изменения

1. Заведите (или возьмите) issue и опишите в нем, что собираетесь делать.
2. Отведите ветку от актуального состояния основной ветки.
3. Внесите изменение **вместе с тестами** (см. ниже) и обновите документацию,
   если меняется поведение или контракт.
4. Откройте Pull Request: в описании — что и зачем, ссылка на issue
   (`Closes #<n>`), как проверяли.
5. Пройдите ревью; CI (сборка, vet, lint, тесты) должен быть зеленым.

Сообщения коммитов — в формате **Conventional Commits**
(`feat:` / `fix:` / `docs:` / `refactor:` / `test:` / `chore:` / `ci:`).

## Стандарты кода

- **Чистая архитектура.** Поток зависимостей: `handler → use-case → domain`;
  `repo`/`clients` — адаптеры, реализующие порты use-case'ов. Domain зависит только
  от stdlib и proto-типов. Бизнес-логика — в use-case'ах, не в handler'ах.
- **Без ORM.** Доступ к БД — sqlc + handwritten pgx.
- **Инварианты — на уровне БД.** Ссылочная целостность и уникальность выражаются
  через FK / UNIQUE / EXCLUDE / CHECK / атомарный CAS, а не software-проверками
  «прочитал → проверил → записал» (они подвержены гонкам).
- **Мутации асинхронны** и возвращают `Operation`; чтения синхронны.
- **Ошибки** — стабильный контракт: корректные gRPC-коды и фиксированные тексты,
  без утечки внутренних деталей БД наружу.
- **Каждый файл** с исходным кодом, схемой, конфигом или скриптом начинается с
  хедера лицензии:

  ```
  Copyright (c) PRO-Robotech
  SPDX-License-Identifier: BUSL-1.1
  ```

- **Комментарии** объясняют, какую задачу решает код и как он работает, — кратко и
  по делу; форматирование — `gofmt`.

## Тесты

Изменение не принимается без тестов в том же PR:

- **unit** — use-case'ы через mock-порты (`internal/apps/.../usecase_test.go`),
  домен, конфиг;
- **integration** — для нового SQL/инварианта: testcontainers Postgres, включая
  конкурентные сценарии для CAS/UNIQUE/EXCLUDE (`internal/repo/*_integration_test.go`);
- **e2e** — для нового публичного RPC: декларативный кейс в `tests/newman/cases/*.py`
  (минимум один happy- и один negative-сценарий), затем `gen.py`.

Тестируйте поведение, а не реализацию. Тесты, проверяющие отсутствие уже удаленного
поведения, в кодовой базе не держим.

## Безопасность

Об уязвимостях сообщайте приватно, не через публичные Issues —
см. [`SECURITY.md`](internal/handler/SECURITY.md).
