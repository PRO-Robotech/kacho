# Как участвовать в разработке kacho-api-gateway

Спасибо за интерес к проекту. Этот документ описывает, как сообщить о проблеме,
подготовить изменение и провести его через ревью.

## Сообщить об ошибке или предложить улучшение

Все задачи ведутся в **GitHub Issues**:
`https://github.com/PRO-Robotech/kacho-api-gateway/issues`.

- **Баг** — приложите версию, шаги воспроизведения, ожидаемое и фактическое
  поведение, релевантные логи. Метка `bug`.
- **Улучшение / фича** — опишите задачу и зачем она нужна (какую проблему решает),
  до обсуждения дизайна. Метка `enhancement`.
- Перед заведением проверьте, нет ли уже похожего issue.

Об уязвимостях сообщайте приватно (GitHub Security Advisories), не через публичные
Issues — gateway это периметр аутентификации и авторизации всей платформы.

## Окружение

- Go 1.25+. Для полного E2E — Docker + kind + helm (поднимается umbrella-стенд).
- Сборка и проверки:

```bash
make build          # bin/api-gateway
make test           # go test ./... -race -cover
make vet
make lint           # golangci-lint
make docker         # образ kacho-api-gateway:dev
make helm-lint      # helm lint deploy/
```

## Поток изменения

1. Заведите (или возьмите) issue и опишите в нём, что собираетесь делать.
2. Отведите ветку от актуального состояния основной ветки.
3. Внесите изменение **вместе с тестами** (см. ниже) и обновите документацию,
   если меняется поведение или контракт.
4. Откройте Pull Request: в описании — что и зачем, ссылка на issue
   (`Closes #<n>`), как проверяли.
5. Пройдите ревью; CI (build, vet, gofmt, `go test -race`, lint, security-сканы)
   должен быть зелёным.

Сообщения коммитов — в формате **Conventional Commits**
(`feat:` / `fix:` / `docs:` / `refactor:` / `test:` / `chore:` / `ci:`).

## Стандарты кода

Gateway — это edge-прокси; его «бизнес-логика» — это транспорт, аутентификация,
авторизация и маршрутизация. Правила:

- **Тонкий транспорт.** Handler/middleware парсят запрос, применяют политику и
  передают дальше; доменной логики ресурсов здесь нет (она в backend-сервисах).
- **AuthN+AuthZ на каждом запросе обоих листенеров.** Внешний TLS-endpoint и
  cluster-internal listener подчиняются одним правилам; internal не освобождён.
  Анонимный доступ в production fail-closed.
- **Identity не подделать.** Клиентские заголовки `x-kacho-principal-*` /
  `x-kacho-token-*` всегда стрипаются на входе и выставляются gateway только после
  валидной аутентификации.
- **Internal-vs-external.** `Internal*`-методы никогда не маршрутизируются на
  внешний endpoint (allowlist + `HasInternalSuffix` + listener-origin).
- **Fail-closed.** Недоступность зависимостей и неопределённость прав трактуются как
  отказ, а не как разрешение; кэши ограничены по размеру и TTL.
- **Без утечки внутренних деталей** наружу в ошибках; стабильные gRPC-коды.
- **Каждый файл** с исходным кодом, конфигом или скриптом начинается с хедера
  лицензии:

  ```
  Copyright (c) PRO-Robotech
  SPDX-License-Identifier: BUSL-1.1
  ```

- **Комментарии** объясняют, какую задачу решает код и как он работает, — кратко и
  по делу; форматирование — `gofmt`.

## Тесты

Изменение не принимается без тестов в том же PR:

- **unit** — middleware (JWT/DPoP/mTLS/authz/cache), allowlist, routing, health —
  через таблицы и mock-порты; обязательно негативные пути (подделанный заголовок,
  невалидный/просроченный токен, alg-confusion, replay, internal-на-external);
- **e2e** — публичный контракт через gateway (Newman, kind+helm-стенд);
- **fuzz** — парсеры пользовательского ввода (DPoP-proof, REST-роутер).

Тестируйте поведение, а не реализацию. Тесты, проверяющие отсутствие уже удалённого
поведения, в кодовой базе не держим.
