# InternalAuthzCacheService

**InternalAuthzCacheService** — единственный собственный (native) proto-контракт гейтвея
(`kacho.cloud.apigateway.v1`). Он даёт `kacho-iam` управлять authz-decision-кэшем гейтвея: при
отзыве прав iam точечно сбрасывает записи кэша именно того subject, чей доступ изменился, — так
отзыв сходится за `< 1 с` вместо ожидания poll-цикла (`~30 с`).

Сервис зарегистрирован **только** на cluster-internal mTLS-listener гейтвея (порт `:9091`) и
**никогда** не публикуется на внешнем TLS REST-mux. Его архитектурная роль и модель доверия
описаны на странице [Internal Authz Cache (архитектура)](/architecture/internal-cache).

:::warning Cluster-internal only
Это не tenant-facing API. Единственный вызывающий — `kacho-iam` `SubjectChangeApplier` (applier
push-drainer'а таблицы `subject_change_outbox`). Клиенты платформы к этому сервису не обращаются.
:::

## Метод

### InvalidateSubject — сбросить записи кэша subject

<ApiOperation method="gRPC" endpoint="kacho.cloud.apigateway.v1.InternalAuthzCacheService/InvalidateSubject">

Сбрасывает записи authz-decision-кэша, привязанные к указанному subject (FGA-prefix). Возвращает
`google.protobuf.Empty`. Освобождён от per-request authz (`permission = "<exempt>"`): сервис живёт
только на internal mTLS-listener, единственный caller — iam по mutual-TLS, поэтому subject-level
authz неприменим.

**Идемпотентность.** Subject без закэшированных записей возвращает `NotFound`, который drainer в
iam маппит в «уже применено» (`ErrAlreadyApplied`) — строка outbox помечается `sent_at` и не
ретраится.

Ответ намеренно пустой: число затронутых записей отдаётся метрикой
`apigw.authz.cache.invalidate.entries` (labels: `event_type`, `subject_type`), а не в теле ответа.

</ApiOperation>

<table>
  <thead><tr><th>Поле</th><th>Тип</th><th>Обяз.</th><th>Описание</th></tr></thead>
  <tbody>
    <tr><td><code>subject</code></td><td>string</td><td><strong>да</strong></td><td>FGA-идентификатор <code>&lt;type&gt;:&lt;id&gt;</code> (напр. <code>user:usr&#95;abc123</code>, <code>service&#95;account:sva&#95;xyz</code>, <code>group:grp&#95;team&#95;a</code>).     <tr><td><code>resource&#95;type</code></td><td>string</td><td>нет</td><td>Подсказка области (scope hint). Сейчас игнорируется — сбрасываются <strong>все</strong> записи subject</td></tr>
    <tr><td><code>resource&#95;id</code></td><td>string</td><td>нет</td><td>Парная подсказка к <code>resource&#95;type</code>; сейчас игнорируется</td></tr>
    <tr><td><code>event&#95;type</code></td><td>string</td><td>нет</td><td>Диагностика (лог + label метрики), на поведение не влияет. Канонические значения: <code>binding&#95;revoke</code> / <code>binding&#95;grant</code> / <code>jit&#95;revoke</code> / <code>bg&#95;revoke</code> / <code>group&#95;member&#95;change</code></td></tr>
  </tbody>
</table>

:::note scope-hint — безопасный upper bound
Пока `resource_type` / `resource_id` не влияют на поведение: гейтвей сбрасывает **все** записи
subject. Это безопасная верхняя граница — переинвалидирует несвязанные ресурсы того же subject, но
никогда не под-инвалидирует. Будущая per-resource реализация сможет сузить сброс до
`(subject, resource_type, resource_id)`.
:::

## Пример (grpcurl, cluster-internal)

Вызов идёт по mutual-TLS на internal listener гейтвея; вне кластера — через port-forward и с
предъявлением client-cert (когда mTLS включён).

<CodeBlock language="bash">
  {dedent`
    grpcurl -d '{
      "subject": "user:usr_abc123",
      "event_type": "binding_revoke"
    }' \\
      api-gateway.kacho.svc.cluster.local:9091 \\
      kacho.cloud.apigateway.v1.InternalAuthzCacheService/InvalidateSubject
  `}
</CodeBlock>

Успех — пустой ответ (`{}`); subject без записей — `NotFound` (для drainer'а это «уже применено»).
